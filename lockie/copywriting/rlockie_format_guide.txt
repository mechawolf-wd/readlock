.rlockie Syntax Reference

Related files:

- readlock_course_copywriting_guide.md — content authority (tone, voice, phase structure, lengths, all writing rules)
- readlock_final_test.txt — rule checklist to run after writing, before shipping

This file is the syntax authority: what declarations exist, what fields they take, what shape the file must have. It is NOT the content authority. For tone, voice, register, phase structure, swipe length, consequence/hint rules, @reflect content, color highlight meanings, and every other "what do I actually write" question, the authority is readlock_course_copywriting_guide.md. Where this file touches content rules, it references the matching guide section in square brackets. If anything in this file ever contradicts the copywriting guide, the copywriting guide wins.

METADATA WRITING RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The content writing guidelines (readlock_course_copywriting_guide.md) cover @text, @question, and other content blocks. Metadata fields like title, description, author, and segment titles are not covered by that prompt, but they must follow the same register.

Write metadata the way a person would describe a book to a friend, not the way a marketing team would pitch it to investors.

BAD:  "Explore the fascinating world of cognitive biases and unlock powerful psychological principles that will transform your design thinking."
GOOD: "How cognitive biases shape the way people interact with designed objects."

BAD:  "A comprehensive deep dive into the art and science of color theory."
GOOD: "Why colors make people feel, and how designers have used that for centuries."

Rules:
- Every rule in readlock_course_copywriting_guide.md Sections 5, 6, 7, and 8 applies to metadata as well (banned verbs, banned openers, banned closers, banned inflators, casual slang, em dashes, triple-anything lists).
- Descriptions state what the course is about. They do not sell it.
- Segment titles are short, plain nouns or noun phrases ("Foundations," "Perception and Decisions"). Not taglines.
- If a sentence could appear in any course about any topic and still make sense, it is too generic. Rewrite it to be specific to this course.

DOCUMENT STRUCTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A .rlockie file has three levels of hierarchy:

    @course          <- exactly once, first line of the file
      @segment       <- one or more, groups packages into chapters
        @package     <- one or more per segment, separates lessons
          @text
          @question
          @true_false    content blocks (one or more per package)
          @estimate
          @quote
          @pause
          @reflect

Every line that starts with @ is a DECLARATION. It must appear on its own line with nothing else on that line.
Everything that follows a declaration (until the next @declaration) is that block's BODY.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
@course
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Must be the very first line. Followed by course metadata as key: value pairs.

    @course

    id: [course identifier]
    title: [course display title]
    author: [author display name]
    description: [one-line description]
    color: [hex color, e.g. #6366f1]
    genres: [comma-separated keys, e.g. design, psychology]
    relevant_for: [comma-separated keys, e.g. ux-designers, product-managers]

Rules:
- All seven fields are required.
- One blank line between @course and the first field.
- description must be a single line.
- color must be a valid hex code.

ID format:
- The id follows the pattern: [type]:[kebab-case-title]-[4-char-hash]
- The type depends on the course source:
    - book:       for courses based on a book (e.g. book:thinking-fast-and-slow-xkfm)
    - topic:      for courses about a standalone topic (e.g. topic:psychology-of-design-abwq)
    - article:    for courses based on an article (e.g. article:cognitive-load-theory-plnz)
- The 4-char hash is randomly generated from lowercase a-z.
- The kebab-case title is a shortened version of the course title, not the full title.
- Example: book:thinking-fast-and-slow-xkfm

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
@segment
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Defines a chapter/section. Body is exactly two lines:

    @segment
    [Segment Title]
    [Symbol - 1 to 3 uppercase characters]

Rules:
- A course must have at least one segment.
- The symbol is a short abbreviation (e.g. "BD" for "Basic Design").
- Every segment must be followed by at least one @package.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
@package
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Separates packages (lessons) within a segment. The lines immediately after @package (before the first content block) are package metadata.

    @package
    [Package display title]
    Free

    @text
    ...

- The first non-empty line after @package is the package display title.
- If the word "Free" appears on its own line, the package is marked as free.
- Both are optional. If no title is provided, a default "Package N" is used.

Rules:
- Every package must contain at least one content block.
- The title and Free flag must come before any @text, @question, or other content declarations.

CONTENT BLOCKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────────────────
@text
────────────────────────────────────────────────────

A reading slide. Each line becomes a separate text segment displayed one at a time.

    @text
    First line shown to the learner.
    Second line shown after the first.
    A third line with a <c:g>highlighted topic term</c:g>.

Line rules (format side):
- Each line is one on-screen segment.
- One sentence per line is the default, up to two sentences allowed when they read as a single beat, three is never allowed.
- Every line, whether one or two sentences, fits in about 19 words total.
  [see copywriting guide: Section 4, Sentence construction, Preferred: one sentence per line / Line length]

Inline color markup:
- <c:g>text</c:g>  -> green. Marks the TOPIC of the current package (the concept, study, book, or effect the package is about).
- <c:r>text</c:r>  -> red. Marks a WRONG thing stated inside the text (a misconception, a deliberately wrong statement the narrator is about to correct). Never on the topic, never on anything correct.
  [see copywriting guide: Section 3, Phase 4, Color highlights]

Image link format:
- image-link: "[search description]"

    @text
    The water cycle is one of the most elegant systems on Earth.
    image-link: "Water cycle diagram"
    Look at the graphic.

Image description rules:
The description is used to search Wikimedia Commons. Write it like a librarian
tagging a photograph, not like a designer describing a mockup.

- Use broad, categorical terms that real photographs and diagrams are tagged with.
- Use the actual subject name, not a creative interpretation of it.
- Prefer nouns over adjectives. "Human eye anatomy diagram" not "detailed beautiful eye illustration."
- Add "diagram" or "photograph" when the type of image matters.
- Ask yourself: "Would a Wikipedia article plausibly use this image?" If yes, good.

GOOD: "Human eye anatomy diagram"
GOOD: "Color wheel primary secondary colors"
GOOD: "Crowd of people in a city street"
GOOD: "Gestalt principles proximity similarity diagram"
GOOD: "Coffee shop menu pricing"

BAD: "A futuristic holographic brain with neural pathways glowing in blue" (does not exist on Wikimedia)
BAD: "Minimal flat illustration of cognitive load theory" (too specific, too stylized)
BAD: "Beautiful sunset over the ocean representing peace" (abstract concept, not searchable)
BAD: "An elegant visualization of the anchoring effect" (no real image matches this)

When an image is included, refer to it in the surrounding text: "Look at the graphic" or
"The diagram shows how..." Never leave an image floating without context.

Rules:
- At least one line of text.
- Color markup is rare: maximum 1 to 2 highlights per WHOLE PACKAGE across both colours combined, not per text block. Most @text blocks will contain zero highlights. The one that names the topic gets the green one.
  [see copywriting guide: Section 3, Phase 4, Color highlights]

────────────────────────────────────────────────────
@question
────────────────────────────────────────────────────

A single-choice or multiple-choice question.

    @question
    [Question text - one line]

    answer: [First answer option]
    consequence: [Feedback shown when this answer is picked]

    answer: [Second answer option]
    consequence: [Feedback shown when this answer is picked]

    answer: [Third answer option]
    consequence: [Feedback shown when this answer is picked]

    correct_answers: [comma-separated 1-based indices, e.g. 1 or 1,3]
    explanation: [Explanation shown after answering]
    hint: [Hint shown before answering]

Rules:
- Question text must be exactly one line.
- Minimum 2 answers, maximum 3.
- Each answer must be immediately followed by its consequence.
- Blank line between each answer+consequence pair.
- correct_answers uses 1-based indexing (1 = first answer).
- Both hint and explanation are required.
- Each consequence is 1 to 3 sentences, each sentence at most about 17 words.
- Each hint is exactly one sentence, at most 17 words.
  [see copywriting guide: Section 3, Phase 3, Writing @question consequences / Writing @question hints]

────────────────────────────────────────────────────
@true_false
────────────────────────────────────────────────────

A statement the learner must judge as true or false.

    @true_false
    [Statement to evaluate]

    true: [Text for the true option]
    consequence: [Feedback if true is chosen]

    false: [Text for the false option]
    consequence: [Feedback if false is chosen]

    correct_answers: [1 for true, 2 for false]
    explanation: [Explanation text]
    hint: [Hint text]

Rules:
- Statement must be exactly one line.
- Exactly one true: and one false: option, in that order.
- true: and false: option text should be descriptive, not just "True" and "False".
- correct_answers is 1 if the statement is true, 2 if false.
- hint is required, and follows the same rules as @question hints (one sentence, ~17 words).
- Consequences follow the same rules as @question consequences (1 to 3 sentences).
- The statement should feel semi-obvious after a moment of thought, not a trap and not dumb.
  [see copywriting guide: Section 3, Phase 5, True/False or Deeper Check]

────────────────────────────────────────────────────
@estimate
────────────────────────────────────────────────────

A percentage estimation question. The learner drags a slider to guess.

    @estimate
    [Question asking for a percentage]
    answer: [correct number, 0-100]
    explanation: [Explanation]
    hint: [Hint text]

Rules:
- Question must be exactly one line.
- answer must be an integer between 0 and 100.
- hint is required.

────────────────────────────────────────────────────
@quote
────────────────────────────────────────────────────

A quote slide.

    @quote
    [The quote text]
    [Author name - or leave empty]

Rules:
- First line is the quote (no quotation marks, the UI adds them).
- Second line is the author attribution. Can be left empty.

────────────────────────────────────────────────────
@pause
────────────────────────────────────────────────────

A breather slide with a nice animation and a single sentence. Gives the reader a beat of rest between denser swipes.

    @pause
    [Single sentence]

Rules:
- Exactly one line, exactly one sentence. Never two.
- The sentence is a small interesting fact, observation, or reframing related to the topic of the package. Not trivia, not a quiz, not a recap, not a motivational line.
- Never starts with "Did you know...", "Fun fact:", or "Pro tip:".
- Zero or one @pause per package. Default spot is right after Phase 4, Topic Reveal.
  [see copywriting guide: Section 3, Optional: @pause swipes]

────────────────────────────────────────────────────
@reflect
────────────────────────────────────────────────────

A reflection prompt. Each line is one thinking point.

    @reflect
    [First thinking point, including one concrete example]
    [Second thinking point, including one concrete example]
    [Third thinking point, including one concrete example]

Rules:
- 2 to 3 thinking points, one per line.
- Each point is about 14 to 16 words.
- Each point contains one concrete example (a situation, an object, a moment, or a type of person). A reflect prompt without an anchor fails.
- No platitudes, no fridge-magnet wisdom.
  [see copywriting guide: Section 3, Phase 6, Reflect]

FORMATTING RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Blank lines are meaningful. They separate logical sections within a block.
2. No trailing whitespace on any line.
3. No tabs. Spaces only.
4. No blank lines inside the body of @quote, @pause, or @estimate.
5. One blank line between the last content block and the next @package or @segment.
6. Properties (answer:, consequence:, correct_answers:, explanation:, hint:, true:, false:) are NOT declarations. They use "key: value" syntax with a colon and space.
